## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                 【Arduino UNO R3】            【出力】
DHT11温湿度センサー ─→┐                         ┌──→ 4桁7セグ（温度/湿度表示）
PIR人感センサー ────→┤  状態管理・判定処理     ├──→ DCモーター（ファン）
IRリモコン受信 ─────→┘  （電源/自動/手動/設定）  └──→ シリアルログ（デバッグ）
```

補足説明（図で表現しきれない場合）：

- 自動運転は「温度 >= 25℃ または 湿度 >= 60%」を満たすとファンを動作。
- 追加機能として、PIRが無検知のときは自動運転を停止する。
- 条件達成後にリモコンで強弱/停止を操作した時点で手動運転へ移行し、電源OFFまで手動運転を維持する。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[自動監視] ──（温度/湿度しきい値超え かつ 人検知あり）──→ [自動送風]
[自動送風] ──（リモコンで強弱/停止を操作）──→ [手動運転]

  │
  └─（設定変更ボタン）→ [しきい値設定] ──（設定完了）──→ [自動監視]

[自動送風] ──（しきい値未満 または 人未検知）────────→ [自動監視]
[手動運転] ──（電源OFFボタン）──→ [電源OFF] ──（電源ONボタン）──→ [自動監視]
[自動監視] ──（電源OFFボタン）──→ [電源OFF]
[自動送風] ──（電源OFFボタン）──→ [電源OFF]


```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 自動監視 | DHT11とPIRを周期取得し、起動条件を判定 | 7セグに温度または湿度を表示、ファン停止 |
| 自動送風 | しきい値条件を満たす間、ファンを自動回転 | 7セグ表示継続、ファンON（既定は中速） |
| 手動運転 | リモコン入力に応じて停止/弱/中/強を切替し、電源OFFまで手動を維持 | 7セグ表示継続、ファン速度を即時変更 |
| しきい値設定 | 温度閾値・湿度閾値をリモコンで変更 | 7セグに設定値を一時表示 |
| 電源OFF | すべての出力を停止し省電力状態 | 7セグ消灯、ファン停止 |

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| DHT11温湿度読み取り | 1000ms | **millis** | DHT11の取得周期制約に合わせるため |
| PIR読み取り | 200ms | **millis** | 人感の反応性を確保するため |
| 7セグ表示更新 | 250ms | **millis** | 表示のちらつき防止と操作応答両立 |
| IRコマンド受付 | 常時 | **millis** | 押下直後に反応する必要があるため |

> [!CAUTION]
> `delay()` を使う場合、その間はボタン入力もセンサー読み取りも**すべて止まります**。

## 3. ハードウェア設計

### 3-1. 部品一覧とピン接続

> ※ デジタルピン: D2〜D13 / アナログピン: A0〜A5
> 特殊用途: D0/D1（シリアル通信・使用禁止）、A4/A5（I2C）、D10〜D13（SPI）

| 部品名 | 型番・種類 | Arduinoピン | 入力 / 出力 | 備考 |
|:--|:--|:--|:--|:--|
| IR受信モジュール | 38kHz IR Receiver | D2 | 入力 | グループ共通部品 |
| DHT11温湿度センサー | DHT11 Module | D3 | 入力 | 1-wireデジタル入力 |
| PIR人感センサー | HC-SR501 | D4 | 入力 | HIGHで人検知 |
| 4桁7セグ表示 | TM1637系 | D5(CLK), D6(DIO) | 出力 | 2線式制御 |
| モータードライバ入力 | NPNトランジスタ or MOSFETゲート | D9(PWM) | 出力 | モーター直接駆動禁止 |
| モーター電源 | 3-6V DC Motor | 外部5V系 | 出力 | 逆起電力対策ダイオード必須 |

> [!CAUTION]
> **ピン競合チェック（必須）**
> - D0/D1 は Serial 通信専用のため使用禁止
> - I2C 機器（LCD等）は A4/A5 を使用
> - 同じピンに複数の部品を繋いでいないか確認

**ピン競合チェック結果：** 競合なし

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| チャタリング | ボタンを押したとき | 前回入力から50ms未満の入力を検出 | デバウンス処理でスキップ |
| 温湿度センサー異常値 | DHT11が読み取り失敗したとき | NaN または 温度<0/温度>50/湿度<20/湿度>90 | その周期は無視し、3回連続失敗でファン停止 + 表示を`Err` |
| 人感センサー誤検知 | 外乱で瞬間HIGHになる | 直近2回連続HIGHで有効判定 | 単発HIGHは無視 |
| リモコン未割当ボタン | 割当外コード受信 | コードテーブル照合で不一致 | 何もしない（ログのみ） |
| モーター過駆動 | 長時間強運転 | 強運転連続時間を計測 | 10分超で中速へ自動降格 |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
- ピン競合はなし（D0/D1未使用、D2-D6/D9等を明確に割当）。
- IRremote・TM1637Display・DHTライブラリの併用は可能。
- delay()は使わず、millis()ベースで周期管理しているため応答性・多重処理に問題なし。
- 状態遷移は「電源OFF→ON」「自動送風→手動運転（手動固定）」など抜けがなく、運転ルールが明確。
