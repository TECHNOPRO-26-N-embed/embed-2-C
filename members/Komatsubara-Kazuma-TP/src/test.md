# リモコン式デジタルロック テスト仕様書

作成日: 2026-05-26  
対象: `remote_digital_lock/ard_code.ino`  
対象ボード: Arduino UNO R3

注意: `kairozu.md` に合わせ、本仕様は4桁7セグ（TM1637）をコメントアウト（未使用）構成で実施する。

---

## 1. 目的

この仕様書は、リモコン式デジタルロックの実装について、以下を確認するためのものです。

- 必須機能（IR受信、パスコード照合、施錠/解錠）が正しく動作する
- 追加機能（LED、ブザー、パスコード変更）が正しく動作する
- 異常系（誤入力、ロック状態、未定義入力）で安全側動作になる
- 状態遷移ログがシリアル出力で追跡できる

---

## 2. テスト環境

### 2.1 ハードウェア

- Arduino UNO R3
- IR受信モジュール（VS1838B）
- サーボモーター（SG90）
- RGB LED（330ohm抵抗 x 3）
- パッシブブザー
- IRリモコン
- 9V1Aアダプタ（Arduino本体給電）

### 2.2 ソフトウェア

- Arduino IDE
- シリアルモニタ: 115200 bps
- 使用ライブラリ: IRremote, Servo

### 2.3 実装前提（初期値）

- 初期パスコード: `1234`
- ロック時間: 30000ms（30秒）
- 自動施錠時間: 180000ms（3分）
- IR重複抑制窓: 150ms

---

## 3. 事前準備手順（物理操作）

1. 電源OFFの状態で配線を確認する。
2. D0/D1に配線していないことを確認する。
3. D2, D8, D9, D10, D11, D12の重複配線がないことを確認する。
4. RGB LEDの各色に330ohm抵抗が直列に入っていることを確認する。
5. サーボ配線（信号D9、電源5V、GND共通）を確認する。
6. Arduino IDEで `ard_code.ino` を開く。
7. ボードを「Arduino UNO」に設定する。
8. スケッチを書き込む。
9. シリアルモニタを開き、通信速度を115200に設定する。
10. 電源投入後、起動ログを確認する。

合格条件:

- 起動時に致命的エラーが発生しない
- サーボが施錠側初期位置になる

---

## 4. 判定ルール

- 各テストケースは「期待結果をすべて満たす」場合に合格
- 1つでも期待結果を満たさない場合は不合格
- 不合格時は「実測値」「再現手順」「シリアルログ」を記録する

---

## 5. テストケース一覧

### TC-01 起動初期化

目的:

- 起動直後に安全側初期化されることを確認する

事前条件:

- 配線済み、書き込み済み

操作手順（物理操作）:

1. Arduinoの電源をOFFにする。
2. 3秒待ってから電源をONにする。
3. LED、サーボ、シリアルモニタを確認する。

期待結果:

- LED消灯
- ブザー停止
- サーボが施錠位置
- シリアルに起動ログと状態ログが出る

---

### TC-02 正常解錠

目的:

- 正しい4桁入力で解錠状態へ遷移することを確認する

事前条件:

- 初期パスコードが`1234`

操作手順（物理操作）:

1. リモコンで `1` `2` `3` `4` を順に押す。
2. LEDとサーボの変化を確認する。
3. サーボ、LED、シリアルログを確認する。

期待結果:

- `IDLE -> INPUT -> VERIFY -> UNLOCKED` の順で状態遷移する
- サーボが解錠位置へ回転する
- LEDが解錠状態表示（緑）になる
- シリアルに`[STATE] ... -> ... @ms=...`形式で遷移ログが出る

---

### TC-03 手動施錠

目的:

- 解錠中に施錠キーで即時施錠できることを確認する

事前条件:

- TC-02完了後、解錠中

操作手順（物理操作）:

1. リモコンの施錠キー（KEY_LOCK）を押す。
2. サーボ、LED、シリアルログを確認する。

期待結果:

- `UNLOCKED -> IDLE` に遷移する
- サーボが施錠位置へ戻る
- ブザー停止状態

---

### TC-04 自動施錠

目的:

- 解錠後3分経過で自動施錠することを確認する

事前条件:

- 解錠状態へ遷移済み

操作手順（物理操作）:

1. TC-02で解錠状態にする。
2. 施錠キーを押さずに3分待つ。
3. 経過中に状態表示を観察する。
4. 3分到達時の動作を確認する。

期待結果:

- 約180000msで `UNLOCKED -> IDLE` に遷移する
- サーボが自動で施錠位置へ戻る
- シリアルに状態遷移ログが出る

---

### TC-05 1回誤入力（エラー表示）

目的:

- 誤入力時にエラー遷移と通知が行われることを確認する

事前条件:

- 待機中

操作手順（物理操作）:

1. 誤った4桁（例: `1` `1` `1` `1`）を入力する。
2. エラー時のLED、ブザーを確認する。
3. 一定時間後に待機へ戻るか確認する。

期待結果:

- `INPUT -> VERIFY -> ERROR` に遷移する
- エラー通知音が鳴る
- 規定時間後 `ERROR -> IDLE` に戻る

---

### TC-06 3回連続誤入力（警告ロック）

目的:

- 3回連続失敗で警告ロックへ遷移することを確認する

事前条件:

- 待機中

操作手順（物理操作）:

1. 誤った4桁を3回連続で入力する。
2. 3回目直後のLED/ブザー状態を確認する。
3. 警告中に数字キーを押して無視されることを確認する。

期待結果:

- 3回目で `VERIFY -> ALERT` に遷移する
- 赤LED点滅、断続ブザーが動作する
- 警告中の入力は無視される

---

### TC-07 警告解除（30秒後復帰）

目的:

- 警告ロックが時間経過で解除されることを確認する

事前条件:

- TC-06でALERT状態

操作手順（物理操作）:

1. ALERT状態になった時刻を記録する。
2. 30秒経過まで待つ。
3. 復帰後に数字キーを押して入力可能か確認する。

期待結果:

- 約30000msで `ALERT -> IDLE` に遷移する
- failedCountがリセットされる
- 数字キー入力が再び有効になる

---

### TC-08 取消キーの桁削除

目的:

- 入力中に取消キーで1桁ずつ削除できることを確認する

事前条件:

- 待機中

操作手順（物理操作）:

1. `1` `2` `3` を入力する。
2. 取消キー（KEY_CANCEL）を1回押す。
3. さらに取消キーを押し、入力を0桁にする。

期待結果:

- 取消1回で末尾1桁のみ削除される
- 入力が0桁になると `INPUT -> IDLE` に戻る
- シリアルログ上で入力と状態遷移が追従する

---

### TC-09 パスコード変更（正常系）

目的:

- 解錠中のみパスコード変更できることを確認する

事前条件:

- TC-02で解錠状態

操作手順（物理操作）:

1. 変更キー（KEY_CHANGE）を押す。
2. 新しい4桁（例: `2` `4` `6` `8`）を入力する。
3. 確定キー（KEY_CONFIRM）を押す。
4. 一度施錠し、次に新パスコードで解錠を試す。

期待結果:

- 変更モードで4桁入力を受け付ける
- 確定後に新パスコードが保存される
- 新パスコードで解錠できる
- 旧パスコードでは解錠できない

---

### TC-10 パスコード変更取消

目的:

- 変更モード中に取消した場合、保存されないことを確認する

事前条件:

- 解錠中

操作手順（物理操作）:

1. 変更キーを押す。
2. 任意の数字を数桁入力する。
3. 取消キーを押す。
4. 施錠後、旧パスコードで解錠できるか確認する。

期待結果:

- 変更内容は破棄される
- 旧パスコードが有効のまま

---

### TC-11 重複入力抑制（長押し）

目的:

- 同一キー長押し時に重複入力が抑制されることを確認する

事前条件:

- 待機中

操作手順（物理操作）:

1. 数字キー`1`を長押しする。
2. LED状態とシリアルログを観察する。
3. シリアルログで受理されたキー回数を確認する。

期待結果:

- 不自然な高速重複入力が抑制される
- シリアルに重複無視ログが出る

---

### TC-12 境界値確認（時間）

目的:

- 時間境界で判定が一貫していることを確認する

事前条件:

- シリアルログが確認できる

操作手順（物理操作）:

1. ALERT状態を作る。
2. 29999ms、30000ms、30001ms付近の状態を記録する。
3. UNLOCKED状態を作る。
4. 179999ms、180000ms、180001ms付近の状態を記録する。

期待結果:

- 30000ms到達でALERT解除
- 180000ms到達で自動施錠
- 前後時刻で逆転現象がない

---

## 6. シリアルログ確認項目

確認する主なログ:

- 状態遷移ログ: `[STATE] <from> -> <to> @ms=<time>`
- 同一状態再設定ログ: `[STATE] stay <state> @ms=<time>`
- 分岐ログ: `[BRANCH] ...`

ログ確認の観点:

- 物理操作とログ時系列が一致している
- 状態飛び（例: INPUTを経ずにVERIFY）がない
- 不正操作時に安全側の遷移が記録される
