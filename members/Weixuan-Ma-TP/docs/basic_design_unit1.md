## 1. システム全体像

### 1-1. ブロック構成（入力 → 処理 → 出力）

> ※ Arduinoを中心に「何が入力で何が出力か」をテキストの図で整理します。

**記入例：**
```
【入力】                 【Arduino UNO R3】            【出力】
タクトスイッチ ─────→┐                         ┌──→ LED（赤・緑）
超音波センサー ─────→┤   状態管理・判定処理    ├──→ パッシブブザー
                      └─────────────────────────┘
```

**↓ 自分のシステムを書いてください**
```
【入力】                 【Arduino UNO R3】            【出力】
赤外線リモコン ─────────→ 赤外線受信モジュール（D2） ─────→┐
  ・LED点灯/点滅ボタン                                           │
  ・ブザー通知ボタン           状態管理・判定処理                ├──→ LED（D9）
  ・停止ボタン                 （信号読み取り / ボタン判定 / 出力制御） └──→ ブザー（D8）



```

補足説明（図で表現しきれない場合）：
赤外線リモコンから送信された信号を赤外線受信モジュールで受信する。  
Arduinoは受信したボタン情報を判定し、LEDの点灯・点滅、ブザー通知、停止処理を行う。  
未登録のリモコン信号を受信した場合は、誤動作を防ぐため状態を変更しない。

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

> ※ ガジェットが「どんな状態を持つか」を定義します。
> この設計が loop() の骨格になります。

**記入例：**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中] ──（センサー検知）──→ [計測中] ──（閾値超え）──→ [警告中]
   ↑                                                            │
   └────────────────────（リセットボタン）────────────────────────┘
```

**↓ 自分の状態遷移を書いてください**
```
[電源ON / 初期化]
  ↓（初期化完了）
[待機中]
  ├─（LED操作ボタン）→ [LED通知中]
  ├─（ブザー操作ボタン）→ [ブザー通知中]
  └─（停止ボタン）→ [停止中]

[LED通知中]
  ├─ LEDを点灯または点滅する
  ├─（ブザー操作ボタン）→ [ブザー通知中]
  └─（停止ボタン）→ [停止中]

[ブザー通知中]
  ├─ ブザーを鳴らす
  ├─ LED点滅と同時動作する場合がある
  └─（停止ボタン）→ [停止中]

[停止中]
  ├─ LEDをOFFにする
  ├─ ブザーをOFFにする
  └─（LED操作ボタン / ブザー操作ボタン）→ 対応する状態へ移行

```

**各状態の意味：**

| 状態名 | この状態のとき、何をしているか | LEDやブザーなど出力の様子 |
|:--|:--|:--|
| 待機中 | 赤外線受信モジュールでリモコン信号を監視し、入力ボタンを判定する | LED・ブザーともにOFF |
| LED通知中 | LED操作ボタンの指示に従って点灯/点滅パターンを実行しながら、他ボタン入力を監視する | LED：点灯または点滅、ブザー：OFF |
| ブザー通知中 | ブザー操作ボタンの指示に従って通知音を出し、必要に応じてLED点滅と同時動作する | ブザー：ON（通知音）、LED：OFFまたは点滅 |
| 停止中 | 安全停止状態として全出力を停止し、次の操作入力を待つ | LED：OFF、ブザー：OFF |

補足説明：
停止ボタンを押した場合は、誤作動時でもすぐに LED とブザーを停止できるようにする。  
LED点滅やブザー制御中でも、`delay()`で処理を止めずにリモコン入力を受け付けられるようにする。  

> [!TIP]
> すべての状態から「抜け出せるルート」があるか確認してください。
> 「このボタンを押せば必ずリセットに戻る」が保証されていますか？

### 2-3. タイミング設計（delay vs millis）

> ※ `delay(N)` はN ms間、**すべての処理**が止まります。
> 複数の処理を並行させたい場合は `millis()` を使います。

| 処理 | 必要な周期 | delay / millis | 判断の理由 |
|:--|:--|:--|:--|
| 赤外線リモコン受信 | 常時 | millis | リモコン入力を常に監視する必要があるため |
| LED点灯・消灯 | 入力時 | millis | ボタン操作後、1秒以内に反応させるため |
| LED点滅 | 500ms程度 | millis | 点滅中でも停止ボタンを受け付けるため |
| ブザー通知 | 入力時または一定周期 | millis | ブザー動作中でもリモコン入力を止めないため |
| 連続入力対策 | 100〜300ms程度 | millis | 同じ信号の連続処理を防ぐため |

> [!CAUTION]
> `delay()` を使うと処理が停止し、その間はリモコン入力を受け付けられません。  
> そのため、LED点滅やブザー制御は `millis()` で実装します。

## 4. 異常系・エラー処理の方針

| 異常ケース | 発生条件 | 検知方法 | 対応方針 |
|:--|:--|:--|:--|
| 未登録のリモコン信号 | 設計していないボタンが押されたとき | 受信コードが登録済みコードと一致しない | 何も動作せず、現在状態を維持する |
| リモコン信号の連続受信 | 同じボタン信号が短時間に連続して入る | 前回受信時刻との差を確認する | 一定時間以内の同一入力は無視する |
| 停止できない誤作動 | LEDやブザーが意図せず動作したとき | 停止ボタン入力を確認する | 停止ボタンでLEDとブザーをOFFにする |
| delayによる入力停止 | delay処理中にリモコン入力ができない | 処理中に入力を受け付けられない | delayは使わずmillisで制御する |
| 配線ミス | LEDやブザーが動作しない | ピン番号、GND、抵抗の接続を確認する | 回路図と実配線を照合する |

## 7. AIレビュー記録

> グループレビューの前に必ず実施してください。

　### Q1: 設計の妥当性確認

> 「この基本設計書をもとに Arduino UNO R3 でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay() の問題点はありますか？」

**AIの回答（要約）：**
D2（赤外線受信）、D8（ブザー）、D9（LED）の割り当てに競合はなく、D0/D1未使用方針も妥当です。D10をLED2個目の候補にする案も問題ありません。状態遷移は4状態で整理され、停止優先の流れになっています。delay()は入力取りこぼしの原因になるため、millis()制御方針は適切です。

**対応した内容：**
3-1でピン競合なしとD0/D1未使用を明記し、1-2で停止優先を確認した。2-3でdelay()を避け、millis()で制御する方針に統一した。

---

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay() を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**
物理ボタンのチャタリング対策より、同一リモコン信号の連続受信を一定時間無視する対策が有効です。delay()使用時は入力停止が発生するため不適切で、時間差判定はmillis()で実装するのが望ましいです。今回はアナログセンサー未使用のため、アナログノイズ対策は必須ではありません。

**対応した内容：**
4章に「連続受信の無視」と「delayによる入力停止」を記載し、2-3でmillis()前提を明確化した。ノイズ対策は赤外線モジュールの配線確認（VCC/GND/信号線）中心で整理した。
